# Support Triage Agent

This directory contains the Python entry point for the HackerRank Orchestrate submission.

## Approach

`main.py` builds one unified retrieval index over all markdown files in `../data`.
It always builds deterministic TF-IDF vectors. If `OPENAI_API_KEY` is present, it also builds cached OpenAI embeddings. If `faiss-cpu` is installed, those embeddings are loaded into a FAISS index for vector search.

The agent resolves the best company context, retrieves the top support documents, computes TF-IDF labels, asks the LLM for `product_area`, `request_type`, and `response`, then compares LLM labels with TF-IDF labels before writing the final output. The `justification` remains code-generated for transparency.

No network calls or API keys are required for the offline TF-IDF fallback.

## Run

From the repository root:

```bash
python code/main.py
```

This reads:

```text
support_tickets/support_tickets.csv
```

and writes:

```text
support_tickets/output.csv
```

For retrieval diagnostics:

```bash
python code/main.py --debug
```

To force offline deterministic mode:

```bash
python code/main.py --retrieval-mode tfidf --disable-llm
```

To enable OpenAI embeddings, FAISS retrieval, and OpenAI LLM responses, install dependencies and set:

```bash
pip install -r code/requirements.txt
set OPENAI_API_KEY=your_key_here
set OPENAI_EMBEDDING_MODEL=text-embedding-3-small
set OPENAI_CHAT_MODEL=gpt-5-mini
python code/main.py --retrieval-mode faiss
```

On macOS/Linux, use `export` instead of `set`.

Embeddings are cached under `../data/index/`, which is ignored by git. The default `--retrieval-mode auto` uses FAISS when available, then hybrid embedding retrieval, then TF-IDF.

To check whether your OpenAI key works without processing all tickets:

```bash
python code/main.py --check-openai
```

If OpenAI quota is unavailable, use OpenRouter for the LLM response/classification step while keeping local TF-IDF retrieval:

```bash
set OPENROUTER_API_KEY=your_key_here
set OPENROUTER_MODEL=google/gemma-4-31b-it:free
python code/main.py --check-openrouter
python code/main.py --retrieval-mode tfidf --llm-provider openrouter
```

OpenRouter is used for `product_area`, `request_type`, and `response`; `justification` remains generated by code. OpenRouter does not replace the embedding endpoint here, so FAISS embedding retrieval still requires an embedding provider. Without one, the agent falls back to TF-IDF retrieval.

Optional paths:

```bash
python code/main.py --input support_tickets/support_tickets.csv --output support_tickets/output.csv --data data
```

## Output Columns

- `status`: `replied` or `escalated`
- `product_area`: support category inferred from retrieved documents
- `request_type`: `product_issue`, `feature_request`, `bug`, or `invalid`
- `response`: user-facing response grounded in retrieved support docs or escalation message
- `justification`: concise routing explanation with retrieval scores and source paths
